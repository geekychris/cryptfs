Kernel-Level Transparent Encryption for Linux

AES-256-GCM • Per-File Keys • Zero App Changes • Per-Process Policies • Multi-Key Support

Inspired by Thales CipherTrust Transparent Encryption (CTE)

The Problem

Data at rest is vulnerable

  • Stolen disks expose all data
  • Decommissioned hardware leaks secrets
  • Cloud storage accessed by unauthorized parties
  • Compliance mandates (HIPAA, PCI-DSS, GDPR) require encryption

Existing solutions fall short

  • Full-disk encryption — all-or-nothing, no per-file control
  • Application-level — requires code changes to every app
  • eCryptfs — limited cipher support, no per-process policies
  • fscrypt — filesystem-specific, not stackable

CryptoFS provides transparent, per-file AES-256-GCM encryption at the VFS layer — no application changes, fine-grained access control, works on any filesystem.

How It Works — Stacked Overlay Filesystem

CryptoFS mounts over an existing directory — even the same path. Apps see no difference.

Overlay Mount

You choose what to encrypt by choosing what to mount over. CryptoFS stacks on top of the existing directory:

# Mount over the SAME path — app config unchanged sudo mount -t cryptofs \ /var/solr/data /var/solr/data # Or use a separate lower dir sudo mount -t cryptofs \ /data/encrypted /mnt/decrypted

Multiple independent mounts:

  • Each mount = separate directory tree
  • Each can have its own key & policies
  • Unmounting one doesn't affect others

Encryption Architecture

Key Hierarchy

Master Key(s) (256-bit, up to 64 in key table) └── wraps → FEK (256-bit, unique per file) └── derives → Nonce (96-bit, per extent)
  • Cipher: AES-256-GCM via kernel Crypto API
  • Key size: 256 bits (32 bytes)
  • Auth tag: 128 bits per 4KB extent
  • Nonce: HMAC-SHA256(FEK, inode ‖ extent_idx)[0:12]
  • FEK wrapping: AES-256-GCM with master key

On-Disk File Format

File Header 128 bytes
Extent 0: Ciphertext 4096 bytes
Extent 0: GCM Auth Tag 16 bytes
Extent 1: Ciphertext 4096 bytes
Extent 1: GCM Auth Tag 16 bytes
… more extents …

Overhead: 128B + 16B per extent ≈ 0.4% for a 1MB file

Demo — Tamper Detection

GCM authentication tags detect any modification to ciphertext

Tamper Test
# Flip one bit in the ciphertext
$ python3 -c "flip_bit(encrypted, offset=200)"

# Attempt to read the tampered file
$ cat /mnt/cryptofs/tamper.bin
cat: /mnt/cryptofs/tamper.bin: I/O error

# Check kernel log
$ dmesg | tail -1
cryptofs: GCM auth tag mismatch on inode 1042 extent 0

# Also detects tag bit-flips
$ python3 -c "flip_bit(encrypted, offset=4224)"
GCM verification failed — tamper detected ✓

Why This Matters

  • Silent corruption impossible — any bit flip is detected
  • Ransomware detection — encrypted files with wrong tags fail to read
  • 128-bit auth tags — per every 4KB extent, not just per file
  • Immediate feedback — I/O error at read time, logged to dmesg + audit

Each 4KB extent is independently authenticated. An attacker cannot:

  • Modify ciphertext without detection
  • Swap extents between files
  • Replay old versions of an extent

Getting Started — 5 Steps

ubuntu — bash
# 1. Build everything
$ make all

# 2. Load the kernel module
$ sudo insmod kernel/cryptofs.ko
[cryptofs] module loaded — AES-256-GCM, extent size 4096

# 3. Start daemon & activate a key
$ sudo cryptofs-keyd --foreground &
$ cryptofs-admin key generate --label my-key
Generated key: a1b2c3d4-e5f6-7890-abcd-ef1234567890
$ cryptofs-admin key unlock a1b2c3d4 && cryptofs-admin key activate a1b2c3d4

# 4. Mount
$ sudo mount -t cryptofs /data/encrypted /mnt/decrypted

# 5. Use it — completely transparent!
$ echo "secret data" > /mnt/decrypted/test.txt
$ cat /mnt/decrypted/test.txt
secret data # ← plaintext
$ xxd /data/encrypted/test.txt | head -2
00000000: 4352 5950 4653 3031 ... # ← ciphertext on disk

Key Storage

  • Master keys stored encrypted on disk with Argon2id KDF
  • Unlocked into daemon memory on demand
  • Activated = pushed to Linux kernel keyring
  • FEKs stored in each file header, wrapped by master key

Access Policies

Fine-grained, per-process control — enforced in kernel space

Policy Types

TypeMatches OnStrength

Default behavior:

  • No policies → all processes have access (PoC mode)
  • With policies → default DENY

Usage Examples

Multi-Key Support

Per-user and per-process key assignments with two access models

Two Access Modes

🔓 Transparent

Admin pre-loads keys into the kernel key table. All authorized processes decrypt automatically.

# Admin loads key into kernel cryptofs-admin key activate a1b2c3d4 # → Stored in kernel key table # → Policy-matched processes use it

🔐 Guarded — User Flow

Each user unlocks their own key. Only their processes can decrypt. Key auto-expires.

# User runs (prompted for passphrase): cryptofs-admin key session-unlock a1b2c3d4 # Behind the scenes: # 1. CLI → daemon: validate passphrase # 2. Daemon → CLI: returns key material # 3. CLI calls add_key("logon", # "cryptofs:<id>", key, SESSION_KEYRING) # 4. cat file.txt → kernel request_key() → ✓ # 5. Key expires after --timeout 3600

Policy Example

# Bind key + mode to a policy rule cryptofs-admin policy add \ --dir /mnt/decrypted \ --type uid --value 1000 \ --key-id a1b2c3d4 \ --access-mode guarded --perm allow

Backward compatible: V1 files use key_id = zeros → default key. No policies → allow-all, default key, transparent mode.

How Does the App Get the Key?

It doesn't. The kernel holds the key and decides per-syscall who sees plaintext.

The Mechanics

Admin runs
cryptofs-admin key activate
↓ master key enters kernel keyring
KERNEL MODULE (cryptofs.ko)
Holds key table + per-file FEKs
On every read()/write() syscall:
→ Inspects the calling process
→ Checks UID, binary path/hash, etc.
→ Decides: decrypt or return ciphertext
↓
ALLOWED PROCESS
Sees plaintext
e.g. Solr (UID 8983)
DENIED PROCESS
Sees ciphertext
e.g. root running cat

What the kernel checks

// Called on EVERY read/write — kernel/file.c rc = cryptofs_policy_check(sbi, inode, &key_id, &mode); // Returns key_id + access_mode for the caller int cryptofs_policy_check(..., u8 *key_id, enum cryptofs_access_mode *mode) { // Inspects UID, GID, binary path/hash, comm // First matching rule → copies key_id + mode // No match → DENY (return ciphertext) } // Resolve key material by access mode key = cryptofs_resolve_key(sbi, key_id, mode); // TRANSPARENT → kernel key table lookup // GUARDED → request_key() from session keyring

Key point: The application has zero awareness of encryption. It never touches a key. The kernel intercepts its syscalls and transparently encrypts or decrypts based on policy.

Real-World Example: Encrypting Solr Indexes

Admin Setup (one-time)

admin@ubuntu — bash
# 1. Install & load CryptoFS
$ sudo modprobe cryptofs

# 2. Start key daemon & create a key
$ sudo systemctl start cryptofs-keyd
$ cryptofs-admin key generate --label solr-prod
Generated: f7e2a1b3
$ cryptofs-admin key unlock f7e2a1b3
$ cryptofs-admin key activate f7e2a1b3

# 3. Mount CryptoFS OVER Solr's data dir (overlay)
# Same source & target path — this is a stacked mount.
# Solr's config stays unchanged; it still reads /var/solr/data.
$ sudo mount -t cryptofs \
/var/solr/data \ # lower dir (ciphertext on disk)
/var/solr/data # same path! (overlay mount)
# 4. Set policy: ONLY the solr user gets plaintext
$ cryptofs-admin policy add \
--dir /var/solr/data \
--type uid --value 8983 \ # solr user
--perm allow

# 5. (Optional) Also allow by Java binary hash
$ SHA=$(sha256sum /usr/lib/jvm/.../java | awk '{print $1}')
$ cryptofs-admin policy add \
--dir /var/solr/data \
--type binary-hash --value $SHA \
--perm allow

# 6. Start Solr — it has NO IDEA encryption exists
$ sudo systemctl start solr

What Happens at Runtime

runtime behavior
# Solr writes an index segment (as user solr)
[solr] write("/var/solr/data/core1/_0.cfs")
→ kernel: policy_check(uid=8983) → ALLOW
→ encrypt → write ciphertext to disk

# Solr reads the index back
[solr] read("/var/solr/data/core1/_0.cfs")
→ kernel: policy_check(uid=8983) → ALLOW
→ read ciphertext → decrypt → return plaintext

# Admin tries to peek (as root, uid=0)
# cat /var/solr/data/core1/_0.cfs
→ kernel: policy_check(uid=0) → DENY
→ returns raw ciphertext (binary gibberish)

# Attacker copies raw files off disk
→ gets only encrypted extents + auth tags
→ useless without master key

Why this works

  • Solr uses unmodified standard file I/O
  • The stacked mount at the same path means zero config changes to Solr
  • Encryption key never leaves kernel memory
  • Even root can be denied by policy
  • Binary-hash policy prevents swapping java for a rogue binary

Performance

Design for Speed

  • Per-inode crypto transforms — allocated once per file, lock-free encrypt/decrypt after init
  • 4KB extent-based — page-aligned, efficient random access without decrypting entire files
  • Hardware AES acceleration — uses AES-NI (x86) or ARMv8 CE automatically via kernel Crypto API
  • Deterministic nonces — no disk I/O for nonce storage
  • Concurrent I/O — no global locks on the encrypt/decrypt hot path
